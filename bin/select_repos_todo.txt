select_repos.py -- known issues at high repo_count
===================================================

At repo_count == 500 there are two problems worth fixing:

1. Rate limiting
   GitHub returns 403 or 429 with a Retry-After header when secondary rate
   limits (burst rate, not just hourly quota) are hit. The current code raises
   immediately on anything other than 404, so a rate-limited run fails hard.

   Two possible fixes (not mutually exclusive):

   a. Retry with backoff: inspect the response status for 403/429, read the
      Retry-After header, sleep for that duration, then retry.

   b. Cache existence in the base repo: store a file in kosli-demo/base (e.g.
      alongside all-repos.json) recording which repos already exist. On each
      run, trust the cache for repos marked as existing and only call the API
      for repos not yet in the cache. Once a repo is created it never needs
      rechecking, so the number of API calls would shrink to near zero over
      time.

2. Speed
   500 sequential HTTPS calls at ~200ms each is ~100 seconds just for the
   existence checks. Fix: parallelise repo_exists() calls using
   concurrent.futures.ThreadPoolExecutor.
